## 1. もとの計画

### 1.1 プロダクトの狙い（requirements.md §1）

SR ツール群 3 部作（sr-query-builder → tiab-review → 本拡張）の 3 作目として、SR の**データ抽出工程**を次の一気通貫フローで支援する:

1. Drive 上の採用論文 PDF + プロトコルから AI が抽出スキーマをドラフト
2. AI が各論文から抽出し、各値に verbatim quote（根拠箇所）を付与
3. PDF.js ビューア上で根拠箇所をハイライト表示
4. 人間が accept / edit / reject / not_reported で最終判定（全判断の監査証跡）
5. CSV エクスポート（study_wide / results_long / audit の 3 形式）

サーバーレス（Sheets = DB、Drive = ファイル実体、BYOK の LLM API）。初学者研究者が**方法論的に妥当な形**（AI 事前抽出 + 人間検証 + automation bias 対策 + 監査証跡）でデータ抽出を完遂できることがゴール。

### 1.2 計画の骨子

- **データ設計**: 追記型 Sheets 15 タブ。データ本体は `StudyData`（wide）+ `ResultsData`（long）+ annotator 軸（`ai` / `human_with_ai` / `human_independent` / `consensus`）。AI 根拠は `Evidence`、判定履歴は `Decisions`。entity_level は study / arm / outcome_result / rob_domain の 4 レベル。
- **技術的中核**: quote アンカリング（正規化 → exact / normalized / fuzzy / failed の段階マッチ。§5）。
- **画面**: S1 プロジェクト選択 〜 S12 裁定 の 12 画面 + Options（S11）。
- **リリース計画**（§7）: MVP（単独プロジェクト・単一レビュアー検証・CSV 3 形式・BYOK）→ P1（二重独立抽出 + 裁定、tiab-review 引き継ぎ、OpenRouter カスタムモデル管理、UI 英語化）→ P2（RevMan 直結、PMC OA XML、日本語論文、表の画像認識）。
- **残実装計画**（[remaining-work-plan.md](remaining-work-plan.md)、2026-07-03）: タスク A（ドキュメント整合）/ B（CI-2）/ C（S11 既定モデル）/ D（抽出精度ベンチマーク Q8）/ E（リリース準備）。

### 2.2 横断基盤

- **Google 層**: OAuth（`chrome.identity`）+ Sheets（batchGet / appendRows / 全列範囲）+ Drive（コピー・フォルダ列挙・共有 `permissions.create`）+ Picker（hosted ページ方式、GitHub Pages 配信）。スコープは `spreadsheets` + `drive.file` のみ。
- **LLM 層**: `LLMProvider` 抽象 + Gemini / OpenRouter / OpenAI 互換エンドポイント。`withRetry`（`Retry-After` 尊重）+ `withThrottle`（RPM 平準化）+ `withLogging`（LLMApiLog + Drive フル payload）。レート制限 tier プリセット + 同時実行数（並列足場）。単価表 + モデルカタログ + コスト概算。
- **抽出パイプライン**: planRun（study 単位バッチ・トークン予算）→ executeRun（複数文書連結・document_index・quote アンカリング・Evidence 生成・partial_failure・Sheets 書き込みバッチ化）→ 検証データ束。
- **信頼性**: オフライン書き込みキュー（Decisions 退避 + 再送）・抽出の耐中断性（2 行プロトコル）・`googleFetch` の 429/503 リトライ。
- **品質体制**: jest 144 suites / 2042 tests・**カバレッジ 100% 強制**・Playwright E2E 64 本（+ axe）・CI-1 / CI-2 稼働。
- **実機確認**: S1〜S10 通し（2026-07-03）・v0.10 フェーズ 3 通し（2026-07-09）・レート制限 tier（PR #30）。
- **既定モデルの根拠**: 実データベンチマーク（`experiments/extraction-benchmark-real`、不眠 SR 10 論文の人手 gold）で `gemini-3.5-flash` を確定（Q8 解決）。事前登録ベンチは凍結保持。
- **リリース準備（タスク E）**: `docs/store/` に提出物一式（プライバシーポリシー・権限説明・スクリーンショット 4 枚）・README のユーザー向けセットアップ手順を用意し、**2026-07-12 に Chrome ウェブストアで v0.1.0 を一般公開**（[掲載ページ](https://chromewebstore.google.com/detail/sr-data-extraction-plugin/ibpbkgffgkmdmflamhadbcfjgfljjgip)。ストア検索から誰でもインストール可）。テスター配布段階のフィードバック（issue #48 等）は解消済み。

## 4. 機能追加の提案（ベストの SR データ抽出・RoB 評価支援アプリへ）

> **注**: 本節は実装担当（Claude）からの**未合意の提案**。既存の設計判断（追記型・サーバーレス・「換算は解析段階に委ねる」等）と整合する形で選んでいるが、採否・優先順はユーザー判断。各項目に〔価値 / コスト〕の粗い見積りを付す。

### A. 方法論・報告の質（SR として論文にするときの強み）

**提案 1: robvis 互換の RoB エクスポート** 〔価値: 高 / コスト: 小〕
RoB 評価は既に `ResultsData` の rob_domain 行として構造化されている。これを robvis（RoB 可視化のデファクト R パッケージ）が読める CSV（Study, D1..D5/D7, Overall, Weight）へ変換する第 4 のエクスポート形式を追加すれば、トラフィックライト図・サマリバーが即座に作れる。「可視化は他ツールに委譲」というスコープ境界（requirements §1.4）を守ったまま、委譲先への**接続コスト**をゼロにする。既存の CSV ビルダー基盤に 1 形式足すだけで済む。

これは↑の 3.3 13と一緒に対応させよう

**提案 2: レビュアー間一致度レポート（κ / 一致率）** 〔価値: 高 / コスト: 中〕
独立二重レビュー（v0.11）で 2 名の human 行と consensus 行が揃うため、項目単位の一致率・Cohen's κ・不一致セル一覧を計算する素材は既に Sheets 上にある。S12 またはダッシュボードに「一致度」カードを置き、CSV でも出せるようにする。PRISMA 2020 / 査読で問われる「抽出の信頼性」を数字で答えられるようになり、**本ツール自体の評価研究（requirements §8）にもそのまま使える**。v1 簡略化で省略した κ の正式実装をここに含める。

issue化

**提案 3: 検証所要時間の計測レポート** 〔価値: 中 / コスト: 小〕
requirements §8 の評価指標「検証所要時間（AI 支援あり vs なし）」は、`Decisions` の timestamp 列から study / セクション単位の所要時間を近似できる（連続判定間隔の中央値ベースなど）。ダッシュボードに載せれば、ユーザーには進捗予測（残り時間の見積り）として、研究としては AI 支援の効率評価として二重に効く。新規データ収集は不要で集計ロジックのみ。

これはアプリ内にはいらない

**提案 4: 登録情報とのアウトカム突き合わせ（selective reporting 支援）** 〔価値: 高（独自性）/ コスト: 大〕
v0.10 の study / document モデルで、1 study が registration ロールの文書（試験登録）を持てるようになった。登録文書から事前指定アウトカムを抽出し、本論文の報告アウトカムと並べて表示すれば、**RoB 2 の Domain 5（報告結果の選択）の判定を一次資料つきで支援**できる。「複数報告文書を統合できる」という本ツールの独自構造が最も活きる機能で、既存 SR ツールにほぼ存在しない差別化点。extract-data のスキーマを流用した専用ミニスキーマ（registration 文書のみ対象）+ 突き合わせビューで実現可能。

issue化

### B. 抽出精度・検証効率（毎日の作業が速く・確実になる）

**提案 5: 決定論的な数値整合性チェック** 〔価値: 高 / コスト: 小〜中〕
LLM を使わない純ロジックで、抽出値の機械的な矛盾を検出して検証画面にバッジ表示する:
- arm 別 n の合計 ≒ total N（±ランダム化後除外の許容幅）
- events ≤ n、% と n/N の再計算一致（丸め許容）
- `ci_lower < mean/estimate < ci_upper`、`q1 ≤ median ≤ q3`、`min ≤ q1`・`q3 ≤ max`
- SD・SE の非負、SE < SD（n>1 のとき）

AI の見落としと人間の見落としは相関しうる（同じ表を読み違える）ため、**第 3 の独立チェック系**として効く。validateAiOutput の隣に置ける純関数群で、jest 100% 文化とも相性がよい。表由来項目の精度リスク（requirements §9）への直接の緩和策。

issue化

**提案 6: 引用メタデータの自動取得（study_label 自動生成）** 〔価値: 中 / コスト: 中〕
現状 study_label は手入力。抽出済みテキストから DOI / PMID を正規表現検出（registration_id 検出と同型）し、「First author (year)」形式のラベル案を提示する。まずは**外部 API なし**（PDF 本文の 1 ページ目から著者・年を LLM ワンショットで推定 or 既存抽出フローに 2 項目追加）で始めれば host permission の追加も不要。CrossRef / PubMed API 連携は権限追加とプライバシー記述の更新が要るため第 2 段階。

これはtiab reviewからのimportとして実現してほしい　ほかのAPI連携はいらない

**提案 7: 検証画面の「同項目横断ビュー」** 〔価値: 中 / コスト: 中〕
現在の検証は study 単位（縦方向）。逆軸の「全 study の `sample_size` を一列で見る」ビューがあると、外れ値・単位の混在（mg vs g、% vs 割合）・表記ゆれを一瞥で発見できる。Covidence 等にもない検証効率の改善で、既存の cellState / focusUnits 基盤の再構成で作れる。まずはダッシュボードからの読み取り専用ビューで十分価値がある。

いらない

### C. 下流工程への接続（メタ解析まで途切れさせない）

**提案 8: メタ解析用の派生列（換算値）のオプション出力** 〔価値: 中 / コスト: 中〕
「SD への換算は解析段階に委ねる」（issue #43 の設計判断）は正しいが、換算式の適用自体は決定論的。エクスポート時のオプションとして SE→SD（√n 倍）、CI→SD（Cochrane §6.5.2）、median/IQR→mean/SD（Wan / Luo / Shi 法）の派生列を**元の報告値と別列 + 換算式名の監査列つき**で出せば、原則（報告値をそのまま保持）を壊さずに下流の手作業を削減できる。results_long.csv の拡張のみで UI 変更は最小。

issue化

**提案 9: R スクリプト雛形の生成** 〔価値: 中 / コスト: 小〕
RevMan 直結（P2）より手前の軽い一手として、results_long.csv を読んで `meta::metabin` / `metacont` に渡すまでの R コード雛形（列マッピング済み・アウトカムごとの forest plot 呼び出し）を S10 で生成・コピーできるようにする。Methods 文案カード(§3.1-5)と同じ「コピー用文案」パターンの拡張であり、初学者ユーザー（§1.5 の想定）がエクスポート後に詰まる最初の壁を除去する。

不要

### D. 運用・スケール（複数人・大規模プロジェクトへの備え）

**提案 10: 書き込み競合の検出（複数人運用の堅牢化）** 〔価値: 中 / コスト: 中〜大〕
独立二重レビューで「同一スプレッドシートに複数人が書く」運用が現実化した。追記型タブは競合に強いが、`StudyData` / `ResultsData` の annotator 行 upsert は read-modify-write のため、同一 annotator が 2 窓・2 端末で開くと後勝ち上書きが起きうる。行 hash か更新カウンタによる楽観ロック（不一致なら再読込を促す）を upsert 系に足す。発生確率は低いが、監査証跡の完全性という本ツールの核心価値を守る保険。

issue化

**提案 11: プロジェクトのバックアップ / スナップショット** 〔価値: 低〜中 / コスト: 小〕
Sheets が唯一の DB のため、ユーザーの誤操作（タブ削除・行編集）が単一障害点。Drive API の `files.copy` でスプレッドシート全体を `backups/` へ日付つき複製するワンボタン（+ 大規模操作前の自動促し）を用意する。既存の copyFile / ensureChildFolder で実装でき、リリース後のサポートコストを下げる。

不要　これはGoogle driveの履歴機能を使ってもらえば十分
