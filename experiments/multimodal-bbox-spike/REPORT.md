- **ステータス**: 実施完了
- **実施日**: 2026-07-11
- **モデル**: `gemini-3.5-flash`（REST `generateContent`。SDK 不使用、`x-goog-api-key` ヘッダでキー送信）
- **入力**: `inputs/07.pdf`（スキャン論文 10 ページ。JBIG2 圧縮の白黒スキャン画像 + 複写スタンプ「Reproduced with permission…」。テキスト層なし）

## 1. 実測で確定した座標規約（H1）

**box_2d = `[ymin, xmin, ymax, xmax]`。画像の左上を原点 (0,0) とし、各座標は画像の高さ・幅に対して 0–1000 に正規化した整数値。**

確認手順: p1 のタイトルだけを問い合わせる PROBE 実行（`PROBE=1 npm run bbox`）で `box_2d=[102,79,175,854]` を取得 → `x_px = xmin/1000*幅px, y_px = ymin/1000*高さpx` で overlay 描画 → 目視でタイトル全体を過不足なく囲んでいることを確認済み（`outputs/overlays/p1.png` 参照。推定どおりの規約で一致、順序の入れ替えは不要だった）。

## 2. 実装上のハマりどころ（Node レンダリング環境固有）

`render-pages.ts` の初回実行では、JBIG2（スキャン画像の圧縮方式）の wasm デコーダが `Ensure that the wasmUrl API parameter is provided` で無音失敗し、**全ページが複写スタンプの1行だけの白紙 PNG になっていた**（テキストや線画が一切描画されない）。原因は pdfjs-dist 6.x の既定 `wasmUrl: "wasm"`（相対パス。ブラウザの `document.baseURI` 前提）が Node の `fs.readFile` ベースの `NodeBinaryDataFactory` には効かないため。`getDocument()` に `wasmUrl` / `cMapUrl` / `standardFontDataUrl` を `node_modules/pdfjs-dist/{wasm,cmaps,standard_fonts}/` の実パス（**末尾 `/` 必須、Windows でも `/` 固定**。`\` だと `getFactoryUrlProp` が `Invalid factory url` で例外）で明示して解消した。

なお pdfjs-dist 6.x は Node 実行時（`isNodeJS` 検出）に内部で `require('@napi-rs/canvas')` して DOMMatrix/Path2D をポリフィルし、`canvas` オプションへ直接 `@napi-rs/canvas` の Canvas インスタンスを渡すだけで描画できる（自前の `canvasFactory`/`canvasContext` 併用は不要）。

**この問題は本スパイクの Node ツール固有**（ブラウザ/拡張本体の pdfjs は `chrome.runtime.getURL` で wasm を含む同梱アセットを解決する別経路）であり、拡張本体には影響しない。ただし今後 Node 側で PDF→画像変換ツール（fixture 生成等）を作る場合は同じ罠を踏むため、この対処法を記録しておく価値がある。

## 3. responseSchema の box_2d に観測されたデータ品質の癖

8 項目中 2 項目（p1 の `first_author` と `final_response_rate`）で、`box_2d` が **5 要素**で返った（例: `[201, 98, 213, 201, 201]`、`[434, 399, 448, 429, 429]`）。いずれも**末尾の要素が直前の要素と重複**しており、末尾 1 要素を落とすと妥当な 4 要素矩形に復元できた（`normalizeBox()` で対応。復元後は目視でも正しい位置を指していた）。`responseSchema` で `box_2d: { type: ARRAY, items: { type: INTEGER } }`（長さ制約なし）を使うと **4 要素固定を保証できない**実例であり、本実装で採用するなら長さ検証 + 末尾重複の復元 fallback（今回の実装のまま）か、`minItems`/`maxItems`（Gemini v1beta の `Schema` オブジェクトがサポート — Fable 監査で API リファレンス確認済み）で長さを 4 に固定した上で、**復元後に 0–1000 範囲・`ymin≤ymax`/`xmin≤xmax` の順序検証も残す両建て**が必要（今回の `normalizeBox()` は「末尾重複 drop」のみで範囲・順序・NaN の検証が無く、中位置重複や退化矩形を誤救済しうる）。

## 5. 表セルと本文の精度差（H4）

本サンプルでは**差が見られなかった**。表セル 4 項目（Table 1 × 2、Table 2 × 2）はいずれも◎判定で、本文・見出し 4 項目（◎2・○2）と同等かそれ以上にタイトだった。p4 は左右 2 段組レイアウトで右列に表があるが、2 段組であること自体が精度に悪影響を与えた様子もない。○判定になった 2 件（first_author, final_response_rate）はいずれも box_2d の 5→4 要素復元を要した項目であり、**表セルではなく本文側**で発生した。座標そのものの粗さというより、responseSchema の癖（§3）に由来する軽微なズレとみられる。

n=1 論文・8 項目という小標本のため統計的な結論ではないが、シグナルとしては「スキャン品質・2段組・小さな表数値であっても box_2d の実用精度は保たれる」という H4 を支持する結果。

## 6. 総合判定: 🟢 Green

- H1: 座標規約は 1 要素の実測で確定・以降 8 項目すべてで矛盾なく再現 → 成立
- H2: 本文・見出し・表セルいずれも対象を実際に囲めている（8/8 命中、外れ 0） → 成立
- H4: スキャン品質（複写・ノイズ）・2段組・小さな表数値でも精度劣化なし → 成立

判定基準（PLAN.md §5）に照らし、本文・表セルとも実用域のため **🟢 Green**。ただし n=1・8 項目のシグナル判定であり、他のスキャン品質（低解像度・傾き・手書き注記混入等）や複雑な表（結合セル・罫線なし表）での再現性は未検証。

**条件（Fable 監査で追加された誠実性の限界）**: 本スパイクはプロンプトに各項目の期待値（「52% のはず」「.95 のはず」等）を与えた **「既知値の位置特定」タスク**であり、本番想定の「**未知値を抽出しつつ位置も返す**」より易しい条件。したがって位置特定精度も本番より楽観側に振れうる。Green は「座標ハイライトが原理的に成立し実装候補になる」ことを示すが、本番プロンプト（値も未知）での位置精度は S6 パイロット相当で別途測る必要がある。

## 7. 拡張本体へ持って行くなら必要な変更メモ

1. **`GeminiProvider` への画像入力経路の追加**: 現状の `GeminiProvider` はテキストのみの `contents.parts` を想定していると思われる。`no_text_layer` 文書向けに `inline_data`（`image/png`, base64）を parts に追加する経路と、`box_2d` を含む responseSchema 拡張が必要。
2. **座標写像式**（PLAN.md §6 のとおり。回転 0 のみ実測済み）:
   ```
   UserSpaceRect {
     x:      xmin / 1000 * W_pt,
     width:  (xmax - xmin) / 1000 * W_pt,
     y:      H_pt * (1 - ymax / 1000),
     height: (ymax - ymin) / 1000 * H_pt,
   }
   ```
   **回転 0 のページに限り**、これを `src/lib/pdf/viewportRect.ts` の `toDisplayRect(rect, { width: W_pt, height: H_pt, rotation: 0 })` に渡すと表示座標に一致する（Fable 監査で手計算・数値スポットチェック済み。`top = ymin/1000·H_pt`）。**回転 ≠ 0 は上式のまま `toDisplayRect` の既存回転分岐に委ねてはいけない**（二重回転になる疑い — box_2d は「回転適用後の描画フレーム」の正規化座標で返るが、`toDisplayRect` は「未回転ユーザー空間 → 表示」の変換のため）。正しくは (a) 表示フレームへ直接 `left = xmin/1000·W_disp, top = ymin/1000·H_disp` と写す（`toDisplayRect` を通さない）か、(b) 逆 viewport 変換で未回転空間へ戻してから渡すかのいずれかを**実測で設計し直す**。本プロジェクトは過去に `/Rotate 90` ページでハイライトずれバグの実績があるため、ここは本実装時の必須設計事項。
3. **box_2d の頑健パース必須**: §3 の「5要素・末尾重複」の実例どおり、`validateAiOutput` 相当の検証層で box_2d の長さ・数値性をチェックし、壊れていれば座標なし（=ハイライト非表示だが値/quote は生かす）にフォールバックする実装が要る。今回の `normalizeBox()` がその叩き台になる。
4. **1ページ1リクエストの構成でよさそう**: 今回は複数項目をまとめて1回の画像アップロードで問い合わせても精度が落ちなかった。本実装でも「ページ画像 + そのページに属する全項目」を1リクエストにまとめるバッチ設計が現実的（`planRun.ts` のバッチ分割方針とも整合しやすい）。
5. **Node 側ツールで同様の PDF→画像変換をする場合**は §2 の wasmUrl 明示を忘れないこと（将来 Node で類似処理を書く際の既知の罠として記録）。

## 8. トークン・実行時間・気づいたリスク

- 実行 3 リクエスト（p1/p4/p5。p1 は座標規約バグ調査のため計 2 回実行、最終集計は最新の1回分のみ）+ PROBE 1 リクエスト
- 合計トークン（最終3リクエストぶん）: 8216 tokens、合計応答時間: 約21秒（画像1枚あたり ~1000 image tokens + thinking tokens 数百〜）
- リスク・限界:
  - n=1 論文・8 項目のみ。他のスキャン品質（傾き・低解像度・手書き）は未検証
  - **プロンプトに期待値を与えた「既知値の位置特定」条件**であり、本番の「未知値抽出＋位置」より易しい（§6 の条件。位置精度は本番プロンプトで再測が必要）
  - box_2d の 5要素バグ（§3）は再現条件が不明。低頻度でも本実装では黙って座標なしにフォールバックする設計が必須。`normalizeBox()` は範囲・順序検証が無く要強化
  - ページ回転（≠0）の写像は未実測かつ `toDisplayRect` 素通しは二重回転の疑い（§7-2）。本実装で要再設計
  - 表のセル結合・罫線なしレイアウト等、より複雑な表構造は本 PDF に無く未検証

## 9. 監査（Fable）による結論と本実装への必須事項

別モデル（Fable）でコード・写像式・実データの整合を監査。**総合判定: 条件付き採用** — 座標写像は回転 0 で数学的に正しく（手計算検証済み）、REPORT の定量主張はすべて `outputs/` の実データで裏付けられ、鍵漏洩なし。オーケストレータ（Opus）の overlay 目視も 8/8 命中を独立確認済みのため、**Green（no_text_layer 文書ハイライトの実装候補化）は採用してよい**。ただし以下を条件とする。

- **コミット前（対応済み）**: 入力 PDF・全ページ PNG・overlay PNG は著作物（複写スタンプ付きスキャン）の再配布回避のため gitignore（`.gitignore` 追加済み）。
- **REPORT 追記（対応済み）**: 「既知値の位置特定」条件である旨を §6 に明記。
- **本実装へ持ち込む際の必須設計**:
  1. 回転 ≠ 0 の写像を `toDisplayRect` 素通しにしない（§7-2。二重回転の再設計）。
  2. box_2d 検証に 0–1000 範囲・順序（`ymin≤ymax`/`xmin≤xmax`）チェックを追加し、失敗時は座標なしフォールバック（§3）。
  3. `responseSchema` に `minItems: 4`/`maxItems: 4` を指定した上でアプリ側検証も残す両建て（§3）。
  4. 「見つからなくても最も近い box を返せ」の強制指示は本番プロンプトでは撤廃（幻覚 box 防止）。
